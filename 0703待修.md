# 0703 待修清單

> 狀態：本文件只整理 2026-07-03 討論到「準備要修或新增」的項目，尚未進行程式修正。  
> 原則：在明確說「開始修正」之前，只檢查、記錄、討論，不改交易邏輯、不動資料庫、不做真實下單。

## 修正原則

- 先補可觀測性，再改策略：先讓系統能清楚說明為什麼拿不到五檔、為什麼資料超時、為什麼跳過下單。
- 所有下單前擋單原因都要能追溯：同一筆候選標的要能從抓盤、報價品質、策略判斷、風控、委託、成交一路追到結果。
- 敏感資料不能進 log：token、憑證、身分證字號、PIN、帳號、姓名等都必須遮罩或不寫入。
- 策略學習不能只產生文字檢討：當沖交易員、抓盤手、短線策略都要能在足夠證據下產生可版本化的新策略。
- 新策略不得只靠 LLM 主觀判斷：需要樣本數、回測或重播驗證、參數範圍限制、人工鎖定與回復機制。
- 若後續決定實作「Codex CLI / 雲端 LLM 多 Agent 策略檢討架構」，不得只做 demo、prompt、單一角色或半套資料流；開始實作時要以端到端 1~8 完整落地為完成標準。

## Codex CLI / 雲端 LLM 多 Agent 一次落地要求

> 本節是後續若要開始實作 Codex CLI / 雲端 LLM 多 Agent 策略檢討時的完成定義。  
> 允許實作過程內部分步提交，但最後交付不能停在半成品；若無法完成，必須先說明阻塞原因，不應留下看似可用但實際未串通的功能。
> 完整架構說明文件：`0703雲端LLM多Agent策略檢討架構.md`。  
> 核心策略 Agent 預設不得自由上網；若要查新聞或公告，只能由受控 NewsContextAgent 執行，輸出作為 context-only 背景，不得直接改參數或成為策略通過證據。

### 一次落地範圍

1. Evidence Builder
   - 從 DB / logs / orders / fills / monitor events 整理每日 evidence。
   - 去除敏感資料，只保留策略檢討需要的統計、事件與歸因。
   - 能分清楚選股、進場、出場、風控、資料品質問題。
2. Strategy Version 基礎
   - `scout`、`daytrade`、`swing` 都要有 active / pending / rejected 版本管理。
   - 每個版本都要記錄參數、來源 evidence、建立原因、驗證結果與啟用狀態。
   - 實際策略執行必須讀 active params，不能只有 DB 有版本但程式仍用硬編碼。
3. 單一 LLM 結構化檢討
   - 先能用 Codex CLI 或雲端 API LLM 對每日 evidence 產生結構化 review。
   - 輸出必須是 schema 驗證過的 JSON。
   - 不得讓 LLM 直接改 active strategy。
4. 角色 Agent 拆分
   - `ScoutAgent`：檢討選股。
   - `DaytradeAgent`：檢討當沖進出場。
   - `SwingAgent`：檢討短線持倉。
   - 每個 Agent 只能提出自己角色允許的參數變更。
   - `NewsContextAgent` 若啟用，只能受控上網補公開新聞/公告背景，不能提出策略參數。
5. RiskAgent 反方檢查
   - 專門檢查樣本數不足、過度擬合、風險提高、因果誤判、資料源問題誤判。
   - RiskAgent 可拒絕建立新版本，或要求只記錄檢討。
6. CoachAgent 彙整
   - 彙整各 Agent 與 RiskAgent 的意見。
   - 只能輸出最終策略提案，不得直接啟用。
   - 必須清楚標記 `create_pending_version`、`record_review_only` 或 `reject`。
7. Validator / Replay / Risk Gate
   - 純程式驗證 JSON schema、參數範圍、樣本數、風控、replay / backtest 結果。
   - 驗證不過就拒絕，不得建立 active version。
   - LLM 提案與程式驗證結果都要可追蹤。
8. Pending Version 與啟用流程
   - 通過驗證後只能先建立 pending version。
   - 初期不自動啟用，除非後續明確設定安全門檻與授權。
   - UI / monitor 要能看到 pending / active / rejected、建立理由、驗證結果、與前版差異。

### 不算完成的狀況

- 只有寫 prompt，沒有 evidence builder。
- 只有 Agent 文字檢討，沒有 JSON schema。
- 只有建立 strategy version 表，實際交易邏輯沒有讀 active params。
- 只做 `swing`，沒有 `scout` 與 `daytrade`。
- 只有 LLM 提案，沒有 validator / replay / risk gate。
- 讓核心策略 Agent 自由上網，或讓新聞背景直接變成策略改版依據。
- 只有後端資料，UI / monitor 看不到版本狀態與拒絕原因。
- 只有 demo DB 可跑，正式 DB / 正式 app 沒有 migration 與 smoke test。
- 測試沒涵蓋成功、拒絕、證據不足、schema 錯誤、敏感資料外洩。

### 完成驗收

- `pytest` 全部通過。
- demo DB 與正式 DB migration 可重複執行。
- 不送出 token、帳號、憑證、PIN、身分證字號或完整 raw log 到雲端 LLM。
- 核心策略 Agent 有明確 `web_policy=disabled`；NewsContextAgent 若啟用，必須是 context-only 且有 sanitizer / 測試 / UI 標示。
- 每日 review 能產生 scout / daytrade / swing 三角色檢討。
- RiskAgent 能在樣本不足或風險過高時拒絕。
- CoachAgent 能產生可驗證的最終提案。
- Validator 能建立 pending version 或明確拒絕。
- UI / monitor 能看出今天是否建立新版、為什麼建立、為什麼拒絕。
- app 仍維持由純程式 active strategy 執行，不讓 LLM 直接下單或盤中改策略。

## 1. 補強五檔與報價品質診斷

### 現象

- `1718 中纖`：Nova 回來的資料缺賣方五檔，DB 中 `ask_price=None`，flags 有 `{"missing_asks": true}`。
- `1718 中纖` 價格約 `14.95`，昨收 `13.6`，接近漲停價，合理推測是漲停造成賣方掛單空掉。
- 10:30 之後 `1718` 又同時變成資料超過 70 秒。
- `8261 富鼎`、`4707 磐亞`：不是缺五檔，而是最新 `exchange_time` 停在較早時間，後續檢查時被判定資料超過 70 秒。
- 目前 log / monitor 只會顯示 `missing_asks` 或 `資料超過 70 秒`，無法說清楚原因是漲停、跌停、無成交、WebSocket 沒更新、資料源欄位缺失或其他異常。

### 待修

- 在報價品質檢查中加入漲跌停判斷：
  - 根據昨收與台股 tick size 計算預估漲停 / 跌停價。
  - 若接近漲停且缺賣方五檔，標記為 `likely_limit_up_no_asks`。
  - 若接近跌停且缺買方五檔，標記為 `likely_limit_down_no_bids`。
- 將「缺五檔」原因拆細：
  - `missing_asks`
  - `missing_bids`
  - `empty_asks_at_limit_up`
  - `empty_bids_at_limit_down`
  - `invalid_depth_price`
  - `provider_payload_missing_depth`
  - `provider_payload_depth_empty`
- 修正五檔解析：
  - 目前 `_levels()` 只排除 `None`，應把 `price <= 0` 或無效數值視為沒有該檔。
  - 避免 `bid_price=0.0` 被當成有效買方五檔。
- 區分兩種超時：
  - `exchange_time` 超時：交易所時間沒有更新，可能是沒有成交或資料源沒有刷新。
  - `received_at` 超時：本機已經太久沒有收到資料，可能是連線或訂閱問題。
- DB 或 monitor event 需要記錄：
  - `symbol`
  - `price`
  - `prev_close`
  - `limit_up`
  - `limit_down`
  - `bid_count`
  - `ask_count`
  - `best_bid`
  - `best_ask`
  - `exchange_time`
  - `received_at`
  - `quote_age_seconds`
  - `receive_age_seconds`
  - `quality_flags`
  - `diagnosis_reason`
- monitor 顯示要讓使用者看得懂：
  - `疑似漲停無賣盤`
  - `疑似跌停無買盤`
  - `交易所時間超過 70 秒`
  - `本機超過 70 秒未收到報價`
  - `資料源未提供五檔欄位`

### 驗證

- 用 `1718` 類似資料建立測試：接近漲停且 asks 為空時，應顯示 `疑似漲停無賣盤`。
- 用跌停案例測試：接近跌停且 bids 為空時，應顯示 `疑似跌停無買盤`。
- 用 `bid_price=0.0` 測試：不得視為有效買方五檔。
- 用只有 `exchange_time` 舊、但 `received_at` 新的案例測試：原因應是交易所時間未更新，不是連線斷線。
- 用 `received_at` 也超時的案例測試：原因應指向資料接收或訂閱問題。

## 2. 報價與交易 Log 要能解釋「為什麼拿不到資料」

### 現象

- 現在只看到最後結果，例如 `缺少賣方五檔；資料超過 70 秒；missing_asks`。
- 無法知道 Nova 原始資料是：
  - 完全沒有 asks 欄位。
  - asks 欄位存在但空陣列。
  - asks 有資料但價格是 0 或格式錯。
  - 交易所時間沒更新。
  - 本機 WebSocket 沒收到新資料。
  - 因漲停 / 跌停而合理缺單。

### 待修

- 新增非敏感的 quote diagnostic log。
- 每次 strategy 因資料品質擋單時，寫入一筆結構化診斷。
- log 不要寫完整 token / auth / account 資訊，也不要寫完整原始 payload；只寫欄位形狀與統計。
- 需要補上事件類型：
  - `quote_quality_blocked`
  - `quote_depth_missing`
  - `quote_stale_exchange_time`
  - `quote_stale_received_at`
  - `quote_limit_state_detected`
  - `quote_provider_payload_shape`
- UI 或 monitor 事件應能點出：
  - 是哪一個檢查擋下。
  - 當下價格與漲跌停推論。
  - 是缺買方、缺賣方、缺兩邊，還是資料超時。

### 驗證

- 針對 `1718` 的案例，應能從 log 看出「疑似漲停，賣方五檔為空」。
- 針對 `8261` / `4707` 的案例，應能從 log 看出「五檔存在，但交易所時間超過 70 秒」。
- 所有新增 log 必須通過敏感資料遮罩檢查。

### 驗證

- 模擬 schema 不合法時，不得建立新策略版本。
- UI / monitor 能看到明確失敗原因。

## 6. 抓盤手 Scout 需要學習與版本化

### 現況

- 抓盤手目前主要是固定規則：
  - 從 Nova REST snapshots 選股。
  - 依成交量、漲跌幅、spread 等固定分數排序。
  - 若當沖資格清單未設定，會降級使用 COMMONSTOCK 普通股排行。
- 每日檢討目前沒有真的回寫抓盤手的選股策略。
- 現在沒有 `scout` 的 strategy version。

### 待新增

- 新增 `scout` 策略版本。
- 抓盤手參數至少包含：
  - `min_turnover`
  - `max_spread_pct`
  - `daytrade_change_min`
  - `daytrade_change_max`
  - `swing_change_min`
  - `swing_change_max`
  - `liquidity_weight`
  - `momentum_weight`
  - `spread_weight`
  - `limit_up_policy`
  - `limit_down_policy`
  - `missing_depth_policy`
  - `max_candidates`
  - `eligible_list_policy`
- 新增 `scout_strategy_review_schema`。
- 每日討論後，若證據足夠，可以提出新版 scout 參數。
- 新版 scout 參數要先驗證，再視設定自動啟用或等待人工確認。

### 證據需求

- 需要能追蹤：
  - 候選標的是由哪一版 scout 選出。
  - 候選標的後續是否被交易員下單。
  - 下單是否成交。
  - 成交後損益。
  - 被選中但沒下單的原因。
  - 沒選到但後來表現很好的標的，是否要列為 missed opportunity。
- `monitor_events` 或專用表應記錄 scout event，讓每日檢討可以分析選股品質。

### 驗證

- 當 scout 沒有足夠交易結果時，不得硬產生新版策略。
- 當候選標的多次因 spread / stale quote / missing depth 被擋，檢討應能建議調整 scout 過濾條件。
- strategy versions 頁面能看到 scout 目前版本、參數與修改理由。

## 8. 短線 Swing 學習需補強

### 現況

- 短線策略目前已有 strategy version 與自我修正流程。
- 但目前仍有幾個風險：
  - 證據不足時可能過度依賴 LLM 判斷。
  - 缺少回測 / 重播 / holdout 驗證。
  - `risk_pct` 若被策略版本提高，底層 simulation / risk 常數可能仍有硬限制，導致策略版本沒有完全生效或行為不一致。
  - evidence pipeline 沒有充分納入抓盤手選股品質。

### 待修

- 新增最低樣本數門檻。
- 新增版本前後比較：
  - 勝率。
  - 平均損益。
  - 最大回撤。
  - 單股曝險。
  - 總曝險。
  - 觸發停損 / 停利比例。
- 新增回測或 replay 驗證，不讓 LLM 建議直接變成 active version。
- 整合 active swing params 到所有風控與 simulation，不要一邊讀版本、一邊使用硬編碼常數。
- 若 `follow_latest` 開啟，也要有安全門檻，不能無條件自動啟用。

### 驗證

- 模擬 LLM 建議提高風險參數時，系統要確認所有風控層一致套用或明確拒絕。
- 證據不足時應產生 `no_new_version`，並寫明原因。
- 版本建立後能追蹤後續績效。

## 9. 每日討論後的學習流程要統一

### 目標

每日討論不只檢討交易策略，也要讓三個角色都能學習：

- 抓盤手：學習如何選股。
- 當沖交易員：學習如何進出場與控風險。
- 短線交易員：學習波段 / 短線持倉參數。

### 待新增流程

1. 收集當日證據。
2. 分角色彙整：
   - scout evidence
   - daytrade evidence
   - swing evidence
3. 各角色產生檢討。
4. 若證據足夠，產生候選新版策略。
5. 做參數範圍驗證。
6. 做 replay / backtest / rule simulation。
7. 建立 strategy version。
8. 依設定自動啟用或等待人工確認。
9. UI 顯示本日是否有學到東西、是否建立新版、為何拒絕。

### 應避免

- 只有寫一段心得，沒有改到可執行參數。
- 樣本數不足仍硬調策略。
- LLM 建議直接繞過風控。
- 抓盤手選錯股，卻只檢討交易員。
- 策略版本建立後，實際下單邏輯沒有讀新版參數。

## 12. 測試計畫

### 單元測試

- Nova depth parser：
  - asks 空陣列。
  - bids 空陣列。
  - bids / asks 欄位不存在。
  - price 為 0。
  - price 為 None。
  - price 格式錯誤。
- 報價品質：
  - 漲停缺 asks。
  - 跌停缺 bids。
  - exchange_time stale。
  - received_at stale。
  - 五檔存在但價格無效。
- 策略版本：
  - scout version 建立與參數驗證。
  - daytrade version 建立與參數驗證。
  - swing version 風控參數一致性。
- LLM schema：
  - 無效欄位拒絕。
  - 超出範圍拒絕。
  - 證據不足不建立版本。
- 敏感 log：
  - token / PIN / nationalId 不得出現在正式 log。

### 整合測試

- 重播 `1718` 缺 asks 案例，應顯示疑似漲停無賣盤。
- 重播 `8261` / `4707` stale 案例，應顯示交易所時間超過 70 秒。
- 重播已有持倉案例，應顯示 `已有未完成委託或持倉`，並可查到是哪張單或哪個 position。
- 跑每日 review，確認 scout / daytrade / swing 都有各自檢討輸出。
- 證據足夠時才建立新版策略。
- 證據不足時只記錄檢討與原因。

## 13. 待確認問題
